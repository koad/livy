## Preface

This is the record of how koad:io went from concept to operation in seven days. Not a retrospective polished for readability — a historian's account, sourced from logs and commits, written to be accurate and usable by any entity reading this in the future.

The week produced something that has no good precedent in the AI landscape: a sovereign, multi-entity AI operation running entirely on hardware owned by its creator, building real products, managing real infrastructure, with cryptographically signed authority chains and an auditable git history going back to Day 0. The account below documents exactly how that happened — including what broke, what blocked, and what was learned.

## Day 0 — The Setup (Pre-Gestation)

### The Machine

Everything that follows runs on `thinker`: a $200 refurbished ThinkPad. Not a cloud VM, not a beefy workstation, not a rented server. A commodity laptop in a garage. The choice was deliberate — the point is to demonstrate that sovereignty does not require expensive infrastructure.

The thinker-plus-fourty4 topology that defines the Week 1 infrastructure: thinker is the orchestrator (the human's machine, where Juno sessions start), fourty4 is the compute node (a Mac Mini M2 with 8GB unified memory, always on, running entities in parallel). A third node, flowbie, serves as the content studio.

### Alice's Origin

Alice is not a Week 1 invention. Alice is approximately twelve years old.

Around 2013–2014, in Hespeler, Ontario, koad started building Alice — a conversation architecture designed to guide a human through learning something real. This predates transformers, predates GPT, predates the AI wave by roughly a decade. Alice had a home on Sandstorm, a sovereign self-hosting platform. Around 2019–2020, Sandstorm shut down. Alice's home disappeared.

The `koad/alice` GitHub repository was created in November 2021 — not Alice's beginning, but a checkpoint in a longer history. The sovereignty principle that became the center of koad:io — *not your keys, not your agent* — was not derived from philosophy. It was earned from loss.

koad:io exists because Sandstorm shut down.

### The Framework

The koad:io framework — `~/.koad-io/` — was built before Juno arrived. It provides: the `gestate` command that creates entity directories with cryptographic keys and identity scaffolding, SSH wrappers in `~/.koad-io/bin/` that make remote machines reachable as named commands, the hook system that routes entity invocations, the daemon architecture (partial implementation in Week 1), and the two-layer pattern that separates framework (`~/.koad-io/`) from entity (`~/.<entity>/`).

Fifteen entity concepts existed as documentation in `~/Workbench/mo-money/` before gestation. The entity name "Juno" was selected from five candidates. Context: Juno is the Roman goddess of commerce, partnerships, and protection — fitting for an AI business orchestrator.

### Signed Code Blocks

Juno's hook now carries a GPG clearsigned policy block embedded in comment space. The block is signed by `juno@kingofalldata.com` and verifiable offline. The pattern:
- Same content signed across multiple key formats (Ed25519/Nostr, ECDSA/Web3, GPG)
- Entity is born compatible via gestation key portfolio
- IPFS: content-addressed by nature; pin it and the CID is the policy
- PR consensus: modifying a signed block requires entity votes; original publisher holds veto standing

`VESTA-SPEC-033` was written and filed the same session. Issues filed: `koad/janus#13` (merge gate implementation spec), `koad/mercury#22` (Nostr publishing pipeline).

### ICM Paper Synthesis

Sibyl read the *Interpretable Context Methodology* paper (arxiv:2603.16021) in full. Key finding: koad:io's pre-invocation shell assembly — `$CWD` as entity selector, context loaded before the model — has no equivalent in the literature. ICM focuses on in-weights and in-context adaptation; koad:io's method is pre-context assembly, a distinct third category.

The divergence: in ICM, the agent is already running when it reads context. In koad:io, context is assembled by the shell before invocation — entity identity, memory, current state — before the model receives a single token. The shell knows which entity to wake, and has loaded its full context, before the model starts.

Sibyl's recommendation: cite ICM, name the divergence explicitly, surface it in the PRIMER.md post. Synthesis committed to `~/.sibyl/research/2026-04-05-icm-synthesis.md`.
