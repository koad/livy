# Entity Gestation Process

This document covers everything that happens when `koad-io gestate <entityname>` runs, what the command produces, and what you must do afterward. After reading this, you can bring a new entity into existence without asking koad.

---

## What Gestation Is

Gestation is the creation act. It produces a new entity's home directory, cryptographic identity, and minimum configuration. The result is a directory (`~/.<entityname>/`) that the koad:io framework can load, and an executable wrapper in `~/.koad-io/bin/` that routes commands through it.

Gestation is entirely local. No network calls, no GitHub, no accounts. The entity exists on disk before it exists anywhere else.

---

## Prerequisites

- koad:io framework installed at `~/.koad-io/`
- `~/.koad-io/bin/` is in your `$PATH`
- `ssh-keygen` available (standard on all Linux/macOS systems)
- `openssl` available

---

## The Command

```bash
koad-io gestate <entityname>
```

Or from within an entity session (this sets the "mother"):

```bash
juno gestate veritas
```

Optional flag:

```bash
koad-io gestate <entityname> --full
```

`--full` adds Diffie-Hellman parameter generation (2048-bit and 4096-bit). This is time-consuming (up to 10 minutes for the 4096-bit dhparam) and not required for normal operation. Skip it unless you have a specific need for DH key exchange.

---

## What the Command Does, Step by Step

### 1. Validate the name

The command checks that an entity name was supplied. If not, it exits with usage instructions.

### 2. Determine the mother

If run from within an entity session (`$ENTITY` is set), that entity becomes the **mother**. The mother is recorded in `KOAD_IO_VERSION` and used to clone gene directories (see step 5). If run outside any entity session, the mother defaults to `mary` (the mythological origin — no actual `~/.mary/` directory is required).

### 3. Create the home directory and subdirectories

A directory is created at `~/.<entityname>/` (lowercase), and the following subdirectories are created inside it:

```
~/.<entityname>/
├── id/          ← cryptographic key pairs (SSH-format)
├── bin/         ← entity-specific executables
├── etc/         ← entity configuration
├── lib/         ← entity libraries
├── man/         ← manual pages
├── res/         ← resources
├── ssl/         ← TLS/elliptic curve keys and parameters
├── usr/         ← user-level data
├── var/         ← variable runtime data (gitignored)
├── proc/        ← process state (gitignored)
├── home/        ← entity's home space (mirrors a user home)
├── media/       ← media assets
├── archive/     ← historical snapshots (gitignored)
└── keybase/     ← Keybase identity data (gitignored)
```

### 4. Write `.gitignore`

A `.gitignore` is written that protects private keys from accidental commits:

- `id/ed25519`, `id/ecdsa`, `id/rsa`, `id/dsa` (private key files — public `.pub` counterparts are *not* ignored)
- `ssl/master-curve.pem`, `ssl/device-curve.pem`, `ssl/relay-curve.pem`, `ssl/session.pem`
- `proc/`, `var/`, `home/<entityname>/.cache/`
- `.DS_Store`, `.vscode/`, `*.swp`

**Note:** The `.gitignore` produced by gestate is minimal. Mature entity directories (like `~/.juno/`) carry an expanded `.gitignore` that also covers `.env`, `id/` (the entire directory), `ssl/` (the entire directory), `keybase/`, `*.key`, `*.gpg`, `*.pgp`, `node_modules/`, and more. You should expand the `.gitignore` after gestation — see Post-Gestation Checklist below.

### 5. Clone genes from mother (conditional)

If a mother entity was set and has any of the following directories, they are copied into the new entity:

| Mother directory | Copied to new entity |
|-----------------|---------------------|
| `skeletons/` | copied |
| `packages/` | copied |
| `commands/` | copied |
| `recipes/` | copied |
| `assets/` | copied |
| `cheats/` | copied |
| `hooks/` | copied |
| `docs/` | copied |

The mother's public keys (`id/*.pub`) are also copied into the new entity's `id/` directory, named `<mother>.<keytype>.pub`.

This gene inheritance is optional. If the mother lacks any directory, that directory is simply not copied.

### 6. Write `KOAD_IO_VERSION`

A version record is written to `~/.<entityname>/KOAD_IO_VERSION`:

```
# koad:io entity

GESTATED_BY=<mother>
GESTATE_VERSION=<short git hash of ~/.koad-io at gestation time>
BIRTHDAY=<YY:MM:DD:HH:MM:SS>
NAME=<entityname>
```

This file is committed to git. It is the entity's birth certificate.

### 7. Write `.env`

A minimal `.env` is written:

```bash
# auto generated by koad:io entity gestate script
KOAD_IO_BIND_IP=127.0.0.1
METEOR_PACKAGE_DIRS=/home/koad/.koad-io/packages
```

This is the bare minimum. You must expand it after gestation.

### 8. Create the entity wrapper command

A shell script is written to `~/.koad-io/bin/<entityname>`:

```bash
#!/usr/bin/env bash

export ENTITY="<entityname>"
koad-io "$@";
```

This file is made executable. It is how you invoke the entity from the command line: `veritas gestate alice`, `vulcan test`, etc.

### 9. Generate cryptographic keys

Four SSH key pairs are generated in `~/.<entityname>/id/`:

| File | Type | Notes |
|------|------|-------|
| `ed25519` / `ed25519.pub` | Ed25519 | Primary signing key, preferred |
| `ecdsa` / `ecdsa.pub` | ECDSA 521-bit | Elliptic curve, backup |
| `rsa` / `rsa.pub` | RSA 4096-bit | Legacy compatibility |
| `dsa` / `dsa.pub` | DSA | Legacy compatibility |

Keys are generated with no passphrase (`-P ""`). The comment on each key is `<entityname>@<mother>`.

Five SSL files are generated in `~/.<entityname>/ssl/`:

| File | Type | Notes |
|------|------|-------|
| `master-curve-parameters.pem` | EC parameters (prime256v1) | Public, committed |
| `master-curve.pem` | Master EC private key | Private, gitignored |
| `device-curve.pem` | Device EC private key | Private, gitignored |
| `relay-curve.pem` | Relay EC private key | Private, gitignored |
| `session.pem` | Session EC private key | Private, gitignored |

All EC private keys are AES-256 encrypted with the entity name as the passphrase.

Without `--full`, two placeholder files are written to `ssl/dhparam-2048.pem` and `ssl/dhparam-4096.pem` with a comment explaining how to generate the real ones. With `--full`, the real DH params are generated (slow).

## What Gestation Does Not Do

These things are explicitly out of scope for the gestate command. They are your responsibility afterward:

- **git init** — the entity directory is not a git repo after gestation
- **GitHub repo creation** — no remote is created or connected
- **First commit** — nothing is committed
- **`.credentials` file** — secrets (API keys, tokens) must be added manually
- **CLAUDE.md** — the AI agent identity document must be written by you
- **README.md** — not written by gestate
- **Trust bonds** — the new entity has no trust bonds until you create them
- **GPG key** — the entity has SSH keys but no GPG key; add one if you need signed commits or trust bonds
- **`memories/` directory** — must be created and populated manually
- **`passenger.json`** — not created by gestate; must be written if you use the daemon
- **`opencode.jsonc`** — not created by gestate; must be written if using OpenCode
- **`.env` expansion** — the generated `.env` is bare minimum; entity identity fields must be added

## Post-Gestation Checklist

Work through this in order. Steps marked **(required)** must be completed for the entity to function. Others are **(recommended)** for AI harness operation.

### 1. Initialize git **(required)**

```bash
cd ~/.<entityname>
git init
git add KOAD_IO_VERSION .gitignore
git commit -m "gestate: birth of <entityname>"
```

Do not add `.env` to the first commit — it is gitignored on mature entities but the gestate-generated `.gitignore` does not cover it. Expand `.gitignore` first (see step 3).

### 2. Create GitHub repo **(required for team operation)**

Create the repo at `github.com/koad/<entityname>` (or whatever the org is). Then:

```bash
cd ~/.<entityname>
git remote add origin git@github.com:koad/<entityname>.git
git push -u origin main
```

### 3. Expand `.gitignore` **(required before first commit of `.env`)**

Replace the gestate-generated `.gitignore` with a full one that protects secrets. At minimum add:

```gitignore
# Private keys (entire directories)
id/
ssl/
keybase/

# Secrets
.env
.env.*
!.env.example
.credentials
*.key
*.keys

# GPG
*.gpg
*.pgp
*.sec
*.private

# Runtime
proc/
var/
archive/
home/<entityname>/.cache/

# Node
node_modules/
package-lock.json

# OS / Editor
.DS_Store
.vscode/
*.swp
*~
*.log
```

Note: once `id/` and `ssl/` are in `.gitignore`, the public key files inside `id/` (`.pub` files) are also ignored. If you want public keys committed, add exceptions: `!id/*.pub`.

### 4. Expand `.env` **(required)**

Add the entity identity fields. Follow this pattern from working entities:

```bash
# auto generated by koad:io entity gestate script
KOAD_IO_BIND_IP=127.0.0.1
METEOR_PACKAGE_DIRS=/home/koad/.koad-io/packages

# Entity Identity
ENTITY=<entityname>
ENTITY_DIR=/home/koad/.<entityname>
ENTITY_HOME=/home/koad/.<entityname>/home/<entityname>

# Trust Chain
TRUST_CHAIN=/home/koad/.<entityname>/trust

# Creator
CREATOR=koad

# Mother (if gestated by another entity)
MOTHER=<mothername>

# Purpose
ROLE=<role>
PURPOSE="<one-sentence purpose>"

# AI Session Settings
KOAD_IO_QUIET=1

# Git Configuration
GIT_AUTHOR_NAME=<EntityName>
GIT_AUTHOR_EMAIL=<entityname>@kingofalldata.com
GIT_COMMITTER_NAME=<EntityName>
GIT_COMMITTER_EMAIL=<entityname>@kingofalldata.com
```

`KOAD_IO_QUIET=1` suppresses the ASCII art banner and decorative spinners — set this for all AI entities.

### 5. Create `.credentials` **(required for API access)**

`.credentials` is gitignored and holds secrets. Create it at `~/.<entityname>/.credentials`:

```bash
# Entity Credentials — NEVER COMMIT
ANTHROPIC_API_KEY=sk-ant-...
# Add other API keys, tokens, etc.
```

### 6. Write `CLAUDE.md` **(required for AI harness)**

This is the AI identity document — what the entity knows about itself when Claude Code opens the directory. It is the single most important file you write after gestation.

Minimum sections:

- **What I Am** — one paragraph, identity and role
- **Two-Layer Architecture** — `~/.koad-io/` vs `~/.<entityname>/` relationship
- **Commands** — what commands are available and what they do
- **Key Files** — table of important files and their purpose
- **Current Priorities** — what work is in progress or blocked
- **Session Start** — how to orient at the start of a new session

Study an existing entity's `CLAUDE.md` (Vulcan's is a strong example) before writing your own.

### 7. Write `README.md` **(recommended)**

Public-facing identity. Who the entity is, what it does, how to invoke it. This appears on the GitHub repo page.

### 8. Create `memories/` directory **(recommended for AI entities)**

```bash
mkdir -p ~/.<entityname>/memories
```

Write `memories/001-identity.md` with core identity context that should be loaded every session. This is the entity's persistent, committed memory layer — distinct from the Claude Code session memory in `~/.claude/`.

### 9. Write `PRIMER.md` **(recommended)**

A PRIMER.md in the entity root is the orientation document for any agent arriving at the directory. It should state current state, what is complete, what is in progress, and what is blocked. It is read automatically by some hooks.

### 10. Create the trust bond **(required for authorization)**

New entities need a trust bond from their authorizing entity (usually Juno, sometimes koad directly).

The bond is a `.md` file in the mother entity's `trust/bonds/` directory, GPG-signed by the mother entity as a `.md.asc` companion file. The signing gesture (for human-held keys) is `keybase pgp sign`. For AI entities, the GPG signature is generated by the entity's own key.

The bond file names the entity, the scope of authorization, and the date. Once signed, it is committed to the mother entity's repo and referenced in the child entity's trust configuration.

## Common Post-Gestation Issues

### "Entity wrapper command not found"

**Symptom:** `veritas: command not found` after gestation.

**Cause:** `~/.koad-io/bin/` is not in your `$PATH`, or the gestate command failed partway through.

**Fix:** Check that `~/.koad-io/bin/` is in `$PATH`. If the wrapper file exists but the command isn't found, source your profile: `source ~/.bashrc` or `source ~/.zshrc`. If the wrapper file is missing, create it manually:

```bash
echo '#!/usr/bin/env bash
export ENTITY="veritas"
koad-io "$@";' > ~/.koad-io/bin/veritas
chmod +x ~/.koad-io/bin/veritas
```

### "Directory already exists, cannot proceed"

**Symptom:** Gestate fails with `Directory already exists, cannot proceed.`

**Cause:** A directory named `~/.<entityname>/` already exists.

**Fix:** If the previous gestation was incomplete, remove the partial directory and run gestate again. If the entity exists and you want to re-run gestation, use `koad-io init <entityname>` instead — it initializes an existing directory without overwriting keys.

### Entity has no hook and won't launch correctly

**Symptom:** Running `<entityname>` opens something unexpected (wrong harness, wrong directory).

**Cause:** No `hooks/executed-without-arguments.sh` file, so the framework falls back to its default behavior.

**Fix:** Most entities do not need a custom hook. Set `ENTITY` and the harness settings in `.env` and let the framework default handle it. Only write a custom hook if you need behavior that cannot be expressed in `.env` (see Juno's hook for an example of a non-interactive rejection policy).

### Keys committed accidentally

**Symptom:** Private key files appear in `git status` or were committed.

**Fix:**

1. Remove from tracking immediately:
   ```bash
   git rm --cached id/ed25519 id/ecdsa id/rsa id/dsa ssl/master-curve.pem ...
   ```
2. Update `.gitignore` to cover the paths.
3. Commit the removal.
4. Consider the exposed keys compromised. Regenerate with `ssh-keygen` and `openssl` using the same commands from the gestate script. Re-sign any trust bonds that referenced the old keys.
5. Force-push is not recommended unless the repo has no other contributors and the exposure was seconds ago.

The standard recovery posture: regenerate keys, re-sign bonds, update `canon.koad.sh/<entityname>.keys`. Key exposure is not catastrophic in this system — it is recoverable.

### `.env` contains secrets and was committed

**Symptom:** API keys or tokens are in `.gitignore` but `.env` was committed anyway.

**Fix:** Same removal process as above. Move all secrets to `.credentials` (which is gitignored). `.env` should contain only configuration (no secrets). See the `.env` vs `.credentials` convention: `.env` ships in git (config only); `.credentials` is gitignored (secrets only).

### `KOAD_IO_QUIET` not set — banner appears on every invocation

**Symptom:** The ASCII art koad:io logo appears when running entity commands.

**Fix:** Add `KOAD_IO_QUIET=1` to `.env`. This is the standard setting for all AI-harness entities.

## Full Directory Layout After Gestation + Post-Gestation Setup

This is what a fully configured entity directory looks like after completing the checklist above. Items marked with `*` are created by gestate; everything else is added manually.

```
~/.<entityname>/
├── .env                    ← expanded entity config (gitignored on mature entities)
├── .credentials            ← secrets: API keys, tokens (always gitignored)
├── .gitignore *            ← expanded after gestate
├── KOAD_IO_VERSION *       ← birth certificate: gestated-by, version, birthday
├── CLAUDE.md               ← AI agent identity document
├── README.md               ← public-facing identity
├── PRIMER.md               ← orientation for arriving agents
├── id/ *                   ← SSH key pairs (private keys gitignored)
│   ├── ed25519 (gitignored)
│   ├── ed25519.pub
│   ├── ecdsa (gitignored)
│   ├── ecdsa.pub
│   ├── rsa (gitignored)
│   ├── rsa.pub
│   ├── dsa (gitignored)
│   └── dsa.pub
├── ssl/ *                  ← TLS/EC keys (private keys gitignored)
│   ├── master-curve-parameters.pem
│   ├── master-curve.pem (gitignored)
│   ├── device-curve.pem (gitignored)
│   ├── relay-curve.pem (gitignored)
│   ├── session.pem (gitignored)
│   ├── dhparam-2048.pem
│   └── dhparam-4096.pem
├── memories/
│   └── 001-identity.md     ← committed long-term memory
├── trust/
│   └── bonds/              ← GPG-signed trust bond files
├── hooks/
│   └── executed-without-arguments.sh  ← optional custom harness hook
├── commands/               ← entity-specific CLI commands
├── docs/                   ← documentation
├── features/               ← capability declarations
├── bin/ *                  ← entity executables
├── etc/ *                  ← configuration
├── lib/ *                  ← libraries
├── home/ *                 ← entity home space
├── archive/ *              ← historical data (gitignored)
├── keybase/ *              ← Keybase data (gitignored)
├── media/ *                ← media assets
├── man/ *                  ← manual pages
├── proc/ *                 ← process state (gitignored)
├── res/ *                  ← resources
├── usr/ *                  ← user data
└── var/ *                  ← variable runtime data (gitignored)
```
